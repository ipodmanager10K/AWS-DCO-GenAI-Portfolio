# [교육용 샘플 데이터] DCO 가상 랙 점검 및 로그 확인 체크리스트 (Rack Checklist)

> [!NOTE]
> **교육용 샘플 데이터 사용 안내**
> 이 문서는 AWS DCO 인턴십 직무 이해를 위한 생성형 AI 활용 및 포트폴리오 제작 과정의 실습 자료입니다. 
> 본 문서에 기재된 모든 정보는 가상의 교육용 샘플 데이터이며, 실제 AWS 내부 정보, 실제 장비명, 실제 IP, 실제 시리얼 번호 및 실제 고객 정보와는 전혀 무관합니다.

---

## 1. 개요
본 체크리스트는 실제 데이터 센터의 물리적 장비를 점검하는 절차가 아닙니다. 교육용 샘플 로그 분석 결과 및 가상의 장애 티켓 내용을 바탕으로 DCO 엔지니어가 기록하고 준수해야 할 핵심 항목들을 정리하는 학습용 체크리스트입니다.

---

## 2. 체크리스트 항목 (Checklist Items)

### [1단계: 로그 및 티켓 정보 확인]
- [ ] **발생 시간 확인:**
  - 샘플 로그와 티켓 내에 이벤트(CRC 에러 증가, Link Down 등)가 발생한 정확한 타임스탬프(발생 시간)를 식별하고 매칭하였는가?
  - 예: `2026-07-07 15:10:00 KST`
- [ ] **샘플 장비명 확인:**
  - 이벤트가 발생한 가상 장비명이 정확히 식별되었는가?
  - 예: `SAMPLE-TOR-SW-01` (접두어가 `SAMPLE-`, `EDU-` 등으로 표기되었는지 검증)
- [ ] **반복 발생 여부 확인:**
  - 해당 장비의 특정 포트(예: `eth-1/1`)에서 CRC 에러 및 링크 단절 이벤트가 이전에도 반복적으로 관찰된 이력이 있는지 샘플 로그 히스토리를 확인하였는가?
- [ ] **Ticket 상태 확인:**
  - 현재 해당 장비에 대해 접수된 가상 장애 티켓의 상태(Open, Pending, Escalated 등)와 티켓 ID(`EDU-TICKET-2026-1007`)가 정확히 연동되어 작동하고 있는지 확인하였는가?

### [2단계: 분석 및 보고서 작성 가이드]
- [ ] **원인 단정 금지 (중요):**
  - 로그에 수집된 가상의 데이터 현상(CRC 에러 증가, Link Down)만을 객관적으로 기록하고, 분석 단계에서 원인(예: "광케이블 단선 확정", "커넥터 오접촉")을 섣부르게 단정 짓지 않고 객관적 현상만 명시하였는가?
- [ ] **보고서 초안 작성 메모 (Handover / Incident Report Draft):**
  - 가상 로그 분석 내용을 인수인계 양식이나 보고서 초안으로 가공하여 정리했는가?
  - 정리 메모 서식 예시:
    * *장비명:* `SAMPLE-TOR-SW-01`
    * *포트:* `eth-1/1`
    * *관찰 사실:* CRC 에러 누적 이후 Link Down 관찰됨.
    * *상위 전달 여부:* HW 점검 및 파트 확인을 위해 Escalation 처리 완료.

---

## 3. 보안 및 비실행 준수 사항 (Security Rules)

1. **실제 정보 포함 금지:** 실제 데이터 센터 내부의 IP, 장비의 고유 시리얼 번호, 실제 고객의 계정 정보 및 티켓 번호를 절대 사용하거나 작성하지 않습니다.
2. **점검 명령 실행 금지:** 외부 대상 또는 실제 운영망 대상의 `ping`, `traceroute` 등 네트워크 진단 명령어를 문서 내에 권장하거나 제안하지 않습니다.
